Android 启动过程简析（二）之 zygote 进程

发表于 2017-11-05 | 分类于 Android进阶 | | 阅读次数

zygote 启动过程

上文已经分析过，zygote 在 app_main.cpp 中的 main 函数启动，但是在 zygote 启动过程中具体做了一些什么工作呢？本文将对此进行分析。

app_main 分析

首先从启动入口进行分析

可以看到在 main 函数中首先创建了一个 AppRuntime 对象，然后就会对传入的参数进行判断，之所以需要判断 – start-system-server 的原因在于 init.zygote64_32.rc 中 32 位进程没有此项参数，之后当带有相应参数的 zygote 被启动的时候会将 startSystemServer 和 zygote 这两个标志位置为 true，然后把所有的参数添加到 args 中，最后通过 AppRuntime 对象启动 zygote。

AndroidRuntime 分析

zygote 通过 AppRuntime 对象进行启动，而 AppRuntime 继承自 AndroidRuntime，所以实际上 zygote 是通过 AndroidRuntime 的 start 函数进行启动

AndroidRuntime 位于 frameworks/base/core/jni

start 函数首先会判断 startSystemServer 是否为 true，如果是的话就会输出启动事件，然后会通过 startVm 函数创建启动 DVM，在启动 DVM 之后会通过 startReg 函数注册 JNI 方法，注册完毕之后会将传入的 options 参数转换为对象数组，完成这一切之后会通过 className 去获取到 jclass 对象，然后获取对应的 main 方法，最终通过 CallStaticVoidMethod 去执行 main 方法。对于 zygote 而言就是调用了 com.android.internal.os.ZygoteInit 的 main 方法。

ZygoteInit 分析

ZygoteInit.java 位于frameworks/base/core/java/com/android/internal/os 下

main 方法中通过 registerZygoteSocket 方法对 socket 进行了注册，注册的 socket 用于和 ActivityManagerService 进行通信，然后调用 preloadByName 对类和资源进行加载，之后调用 startSystemServer 创建 SystemServer 进程，最后通过 runSelectLoop 等待来自 ActivityManagerService 的请求。 这里最为关键的是 socket 以及 SystemServer 进程，所以我们需要进一步了解这两个部分。

registerZygoteSocket

首先是注册 Socket 部分， registerZygoteSocket 主要是利用文件描述符创建了一个 LocalServerSocket 对象而已。

startSystemServer

startSystemServer 主要工作是利用设置的参数通过 forkSystemServer 创建 SystemServer 进程，然后调用 handleSystemServerProcess 来启动 SystemServer 进程。

runSelectLoop

runSelectLoop 的主要工作是获取 registerZygoteSocket 中注册的 Socket 的文件描述符保存在 fds 中，然后对 fds 进行遍历，将所有信息放在 StructPollfd 数组中，对数组进行循环遍历，如果 i 为 0 说明有新的连接请求到来，然后会通过 peers.get(i) 获取到 Socket，之后执行 runOnce 创建新的应用程序进程。

总结

  1. Zygote 进程的启动过程主要工作是创建启动 DVM、注册 JNI 方法、注册用于处理 ActivityManagerService 连接的 Socket、创建启动 SystemServer 进程、预加载类资源
